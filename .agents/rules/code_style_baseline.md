# Code Style Baseline

Engineering standards that apply to all code written or modified in this project.

---

## Readability First

**Readability is the primary engineering standard.**

Prefer implementations that are clear, direct, and easy to maintain over extra layering
introduced for theoretical completeness, future reuse, or stylistic symmetry.

### What this means in practice

- If the current requirement only needs a simple structure, do not split it into extra
  managers, controllers, helpers, wrappers, state machines, or abstract base classes.
- When stronger abstraction conflicts with better readability, default to better readability.
- When theoretical generality conflicts with the current business need, default to the
  current business need.

### Extension points

Extension points are allowed only when **all** of the following are true:
- The current responsibility boundary is already clear.
- The interface is likely to be used in the near term (not hypothetically).
- The added indirection does not materially reduce readability.

### New code priorities

New code should optimize for, in this order:
1. Fast comprehension by other engineers reading it cold.
2. Low follow-up modification cost.
3. Direct debugging paths — fewer hops from symptom to root cause.

### Anti-patterns to avoid

- Splitting a simple operation into a helper for "reuse" that is never reused.
- Adding a configuration flag for a variation that only exists hypothetically.
- Wrapping a third-party API in an abstraction layer before the API contract is understood.
- Designing for uncertain future changes that have not been confirmed as requirements.
- Three similar lines of code is better than a premature abstraction.

## No Speculative Changes

Do not add features, refactor code, or make "improvements" beyond what was asked.

- A bug fix does not need surrounding code cleaned up.
- A simple feature does not need extra configurability.
- Do not add docstrings, comments, or type annotations to code you did not change.
- Only add comments where the logic is not self-evident.

Do not add error handling, fallbacks, or validation for scenarios that cannot happen.
Trust internal code and framework guarantees. Only validate at system boundaries
(user input, external APIs, network responses).

## Text Encoding Safety

This rule applies whenever editing files that may contain non-ASCII content
(Chinese characters, em-dashes, mixed-language text, special punctuation).

- Treat all repository text files as **UTF-8 files that must preserve their original bytes**.
- Do not rewrite an entire file through ad hoc shell text pipelines (PowerShell, sed, awk)
  when the file contains non-ASCII content, unless you have verified the write path is
  byte-safe for UTF-8.
- **Prefer localized edits** (apply_patch or equivalent tools) for existing text files.
  If a full-file rewrite is unavoidable, use a tool that explicitly reads and writes UTF-8,
  then verify the result with a UTF-8-aware reader.
- Do not trust terminal rendering alone when checking for encoding corruption.
  PowerShell and some terminals can display UTF-8 correctly while writing corrupted bytes.
  Verify with a byte-safe read if the file contains non-ASCII content.
- If an encoding repair requires a temporary file, **remove the temporary file** before
  finishing the task.

## Auto-Generated Files — Never Modify Manually

Some directories contain files that are entirely auto-generated by export tools, code
generators, or build pipelines. Manually editing these files will cause the changes to be
overwritten on the next generation pass.

## Backwards Compatibility

Avoid backwards-compatibility hacks unless the project has an explicit policy requiring them:
- Do not rename variables to `_old` or `_unused` when removing them — delete them.
- Do not add `// removed` comments for deleted code — the version history has it.
- Do not re-export types from new locations just to avoid updating import paths — update them.
- Do not add feature flags for gradual rollout unless the deployment process requires it.
